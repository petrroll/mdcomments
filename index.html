mdcomments

Native comments for Markdown — structured, portable, and built from existing syntax.

A lightweight comment format that supports anchoring, authorship, timestamps, threading, and metadata — using only standard footnotes, blockquotes, and @-mentions.

Why mdcomments?

Markdown is rapidly becoming the universal format for LLM-generated and LLM-consumed content. Yet there is no portable way to annotate it with structured comments. Existing solutions are either unstructured (HTML comments, %%) or platform-locked (HackMD, Notion, sidecar JSON).

Comments should live in the file and survive round-tripping through any tool.

Design Goals

Anchored

Comments attach to specific text, not just float in the file.

Attributed

Each comment carries an author and timestamp.

Threaded

Replies form a conversation, not a flat list.

Graceful Degradation

In any renderer, comments appear as sensible footnotes — nothing breaks.

Portable

Works in any standard Markdown parser without a plugin.

Survives Round-tripping

Comments aren't stripped on export or import.

How It Works

Inline [^c-id] footnote markers anchor comments to specific text. Comment threads are defined at the bottom of the document using standard footnote syntax, with @author (date): lines and blockquoted messages: 

Revenue grew by 15%[^c-rev1] last quarter.

[^c-rev1]:
    @alice (2026-02-10):
    > Is this YoY?

    @bob (2026-02-11):
    > Yes. Added a clarifying note.

In any standard Markdown renderer, comments degrade gracefully to ordinary footnotes — nothing breaks, nothing is lost.

Why Footnotes?

  1. Already standardized — supported by GitHub Flavored Markdown, Pandoc, Hugo, Obsidian, and most CommonMark extensions.
  2. Graceful degradation — in renderers that support footnotes, comments appear as numbered references. In those that don't, the raw [^c-id] is small and unintrusive.
  3. Natural separation of concerns — a tiny inline marker with a longer definition elsewhere keeps prose clean.
  4. No new syntax — combines three existing primitives: footnotes, blockquotes, and @-mentions.
  5. Convention over specification — the c- prefix and @author (date): pattern are conventions layered on valid Markdown.

Alternatives Considered

  • HTML Comments <!-- --> Universal support, but no anchoring, author, or threading. Too primitive for collaborative review.
  • Reference-Link Hack Works in most parsers but has no structure at all. A clever trick, not a foundation.
  • Obsidian %% Simple syntax, but non-standard and Obsidian-only. Locks you into one tool.
  • CriticMarkup Closest prior art — inline and anchored — but lacks threading and {} collides with attribute syntax.
  • kramdown {::comment} Explicitly a comment directive, but kramdown-only. No anchoring or structure.
  • MDX / JSX Comments Native to React/JSX ecosystem only. Not portable Markdown.
  • Pandoc Footnotes Reference-style and inline forms. Concise but render visibly and lack identity or threading.
  • Marker / PyMdown Highlight Syntax Visual anchoring via {== ==}, but same structural shortcomings as CriticMarkup.
  • Sidecar JSON Files Keeps Markdown pristine but comments are lost if the sidecar is missing. Fragile and non-portable.
  • Platform Databases (HackMD, Notion) Great UX, zero portability. Comments vanish when you export or leave the platform.
